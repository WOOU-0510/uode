# 커밋 메시지 작성 가이드 (uode)

이 문서는 이 저장소에서 **커밋 메시지를 어떻게 쓸지**에 대한 규칙/예시를 정리합니다.  
목표는 “멋진 문장”이 아니라 **변경 의도(왜)와 영향 범위(어디)**를 빠르게 파악하게 하는 것입니다.

## 핵심 원칙

- **한국어로 작성**한다.
- 제목(1줄)만 봐도 **무엇을/왜** 했는지 알 수 있어야 한다.
- “무엇을 바꿨나”보다 **왜 바꿨나(의도/근거/제약)**를 우선한다.
- 커밋은 가능하면 **작게, 하나의 의도**만 담는다.
- 자동 생성된 영문 요약을 그대로 쓰지 말고, 이 가이드 기준으로 **사람이 다듬는다**.

## 형식(권장)

아래 형태를 권장한다.

```
<type>: <한 줄 요약>

<선택> 배경/의도 (왜)
<선택> 변경 요약 (핵심만)
<선택> 영향/주의사항
<선택> 테스트 방법
```

### 제목(한 줄) 규칙

- **50자 내외** 권장(넘어도 되지만, 너무 길면 분리 고려)
- 문장부호 과다 사용 금지
- “수정함/변경함/업데이트” 같은 의미 없는 종결은 지양하고, **구체 동사**를 사용

## 유형(권장 접두사)

팀/개인 로그에서 검색하기 쉽게, 제목 앞에 아래 접두사를 붙이는 것을 권장한다.

- `feat`: 사용자 가치/동작 추가
- `fix`: 버그/오동작/회귀 수정
- `improve`: 성능/UX/개발경험 개선(동작 변경 포함 가능)
- `refactor`: 동작 동일, 구조/가독성/중복 제거 중심
- `docs`: 문서 추가/갱신
- `style`: 포맷/네이밍/비동작 변경(린트 포함)
- `test`: 테스트 추가/수정
- `build`: 빌드 스크립트/의존성/툴링 변경
- `ci`: CI 파이프라인 변경

> 주의: `refactor`는 “동작이 그대로”일 때만 사용한다. 동작이 바뀌면 `improve`/`fix`를 사용한다.

## 내용에 무엇을 써야 하나

### 반드시 포함하면 좋은 정보

- **변경의 이유**: 왜 이 변경이 필요했는지(문제/제약/의도)
- **영향 범위**: 어떤 영역(라우트/레이어/모듈)이 바뀌는지
- **핵심 결정**: 여러 선택지 중 왜 이걸 택했는지(짧게)

### 생략해도 되는 정보

- 파일 목록 나열(필요하면 diff로 충분)
- “코드 정리”, “조금 수정” 같은 모호한 표현

## 예시

### 좋은 제목 예시

- `docs: 프로젝트 구조/상태관리 기준 정리`
- `improve: playground 내비를 라우트 내부 config로 코로케이션`
- `fix: Tauri 런타임이 아닐 때 invoke 에러 메시지 통일`
- `refactor: icon playground 스토어 모듈 확장자 ts로 정리(JSX 제거)`

### 나쁜 제목 예시

- `update`
- `fix`
- `커밋`
- `작업함`
- `refactor`(하지만 실제로는 동작 변경이 포함됨)

## 본문(Body) 작성 가이드

본문은 선택이지만, 아래 중 하나라도 해당하면 **1~5줄**이라도 적는 것을 권장한다.

- 동작 변경이 있다
- 나중에 까먹기 쉬운 결정/트레이드오프가 있다
- 실험용 코드(playground)라서 “왜 유지/왜 예시로 남김”이 중요하다

권장 예시:

```
improve: playground 내비를 라우트 내부로 이동

playground에서만 쓰는 내비 정의/활성화 유틸을 app/(dev)/playground/config로 코로케이션했다.
재사용 가능성이 낮아 shared로 올리지 않고, 라우트 변경 시 영향 범위를 제한한다.

테스트: /playground 사이드바 활성화 확인
```

## 테스트 문구(선택)

가능하면 아래 중 하나를 간단히 적는다.

- `테스트: npm run build`
- `테스트: /playground/tauri-api/health 섹션 동작 확인`
- `테스트: 미실행 (문서만 변경)`

## Squash/merge를 고려한 메시지

PR에서 squash할 가능성이 높다면, 각 커밋 메시지도 “나중에 히스토리로 남아도 의미 있는가” 기준으로 작성한다.  
중간 커밋(실험/되돌림)은 로컬에서 정리하거나, 최소한 제목에 의도를 드러낸다.

## 금지/주의 패턴

- `WIP`, `tmp` 같은 임시 메시지를 main 히스토리에 남기지 않기
- 자동 생성된 장문의 영어 요약을 그대로 붙여넣기
- 한 커밋에 서로 다른 의도(예: UI 개선 + 스토어 리팩터 + 문서)를 과도하게 묶기

## 이 저장소 맥락에서의 추가 규칙

- `playground`는 **예시/학습 공간**이다. “현실적인 구현”과 “패턴 데모” 중 무엇을 택했는지 본문에 남기면 좋다.
- 구조/규칙 관련 변경은 `docs/architecture.md` 갱신을 함께 고려한다.
